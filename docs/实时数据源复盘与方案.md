# 实时数据源复盘与方案

更新时间：2026-06-16

## 本次复盘结论

这次反复出现“默认是假数据、刷新后又重复计算”的根因不是某一个球员写错，而是数据架构里仍然允许本地 JSON 手工文件默认进入线上事实链路。

已调整为：

- 线上默认不再读取 `matches.live.json` 和 `matches.public-results.json`。
- `LOCAL_MATCH_FEED_ENABLED=false` 为默认值，本地 JSON 只能用于开发调试或离线测试。
- 比分、赛程、进球、红黄牌优先来自自动数据源。
- 球员射手榜、助攻榜、红黄牌榜优先来自供应商球员统计接口。
- 没有供应商 Key 时，助攻榜不会用手工数据或预测数据补假结果，而是明确显示“供应商球员榜未接入/仅统计自动比赛事件中包含的数据”。

## 数据源分层

### 1. 正式主源：授权足球数据 API

长期稳定方案必须接入授权数据 API，尤其是助攻、阵容、红黄牌、停赛、技术统计、球员榜这类字段。公开比分源通常不保证这些字段完整。

推荐顺序：

1. Opta / Stats Perform
   - 适合正式商业化产品。
   - 覆盖 live scores、events、lineups、player statistics、xG、深度战术和预测数据。
   - 成本和接入门槛最高，但权威性和稳定性最好。
   - 参考：https://www.statsperform.com/products/opta-data/ 和 https://www.statsperform.com/products/opta-data-feeds/
2. Sportmonks Football API
   - 适合中小型产品落地。
   - 覆盖 fixtures、livescores、events、lineups、statistics、topscorers、injuries、predictions 等模块。
   - 参考：https://www.sportmonks.com/football-api/ 和 https://docs.sportmonks.com/v3/
3. API-Football / API-Sports
   - 当前项目已预留并实现 API-Football 适配器。
   - 用于 fixtures、fixture events、fixture lineups、fixture statistics、topscorers、topassists、topyellowcards、topredcards。
   - 参考：https://www.api-football.com/ 和 https://www.api-football.com/documentation-v3

### 2. 自动备源：ESPN Scoreboard

ESPN Scoreboard 继续作为公开自动备源，用于：

- 赛程状态。
- 已完赛比分。
- 进球事件。
- 红黄牌事件。

限制：

- 当前 ESPN 事件明细通常能给到进球球员，但助攻字段不稳定。
- ESPN 不应作为助攻榜、阵容、停赛、技术统计的唯一权威源。
- 当 API-Football/Sportmonks/Opta 未配置时，助攻榜为空是正确行为，不应再用手工文件补假数据。

### 3. 本地 JSON：仅限开发调试

以下文件保留，但默认不会参与线上实时数据：

- `backend/data/matches.live.json`
- `backend/data/matches.public-results.json`

启用条件：

```env
LOCAL_MATCH_FEED_ENABLED=true
```

该开关只用于开发、离线测试或供应商接口不可用时的人工排查，不应作为线上常规数据源。

## 当前项目实现

新增：

- `backend/services/api_football.py`
  - `fetch_api_football_match_feed()`
  - `fetch_api_football_leaderboards()`
  - `normalise_api_football_leaderboards()`

更新：

- `backend/services/live_match_feed.py`
  - 默认忽略本地 JSON 文件。
  - 支持 API-Football 自动比赛源。
  - 数据优先级：ESPN 自动源 < 显式远程 feed < API-Football。
- `backend/routers/players.py`
  - 配置 API-Football Key 时，射手榜、助攻榜、红黄牌榜直接使用供应商球员统计接口。
  - 未配置 Key 时，只统计自动比赛事件里的事实数据，不使用手工数据。

## 部署配置

QNAP/Unraid `.env` 增加：

```env
LOCAL_MATCH_FEED_ENABLED=false
API_FOOTBALL_ENABLED=true
API_FOOTBALL_KEY=
API_FOOTBALL_LEAGUE_ID=1
API_FOOTBALL_SEASON=2026
API_FOOTBALL_DETAIL_DAYS_BACK=7
API_FOOTBALL_DETAIL_DAYS_FORWARD=1
```

拿到 API-Football Key 后，只需要填入 `API_FOOTBALL_KEY` 并重启后端容器。不要把 Key 写入公开文档或代码仓库。

## 验收口径

每次修改实时数据链路后必须验证：

- 后端测试通过。
- 前端 production build 通过。
- `/api/v1/matches/` 返回 104 场。
- `/api/v1/live/status` 显示当前数据源、命中场次数、是否启用本地文件。
- `/api/v1/players/leaderboards/2026` 的 `live_status.player_stats_source` 与实际来源一致。
- 默认部署环境下 `local_match_feed_enabled=false`。
- 未配置供应商 Key 时，助攻榜不得由手工 JSON 或预测数据补齐。
- 配置供应商 Key 后，助攻榜应来自 `api_football` 或后续接入的 `sportmonks/opta`。
